> Date: 2026-05-10

## Context

SovereignFlow adopted runtime-guard for pytest memory monitoring.
The development environment is WSL2 + VS Code, where VS Code node processes
hold **10–12 GB RAM** at idle, causing the default `tight` thresholds
(min 2048 MB available, max 85% swap) to fire for the entire test run and
skip all 1651 tests — even though pytest itself is not the pressure source.

### Local implementation (SovereignFlow `tests/conftest.py`)

The following pattern was needed to work around this:

```python
import os
os.environ.setdefault("SOVEREIGNFLOW_GUARD_POSTURE", "ci")

_guard = make_pytest_guard(repo_name="SovereignFlow", ...)

def pytest_sessionstart(session):
    _guard.oom_protect(score=-500)
    _guard.preflight_check(abort_on_critical=False, auto_intervene=True)
    _guard.start_background_check(interval_s=60)

def pytest_runtest_setup(item):
    _guard.check_and_log(stage=item.nodeid)

def pytest_sessionfinish(session, exitstatus):
    _guard.stop_background_check()
```

### Validation

All 10 performance baseline tests pass after this change:
```
10 passed in 9.48s
```

## Suggested Improvements

### 1. Add `posture` parameter to `make_pytest_guard()`

**Problem:** `make_pytest_guard()` does not accept a `posture=` argument.
Users must set `<PREFIX>_POSTURE` via `os.environ.setdefault()` before calling
the factory — this is non-obvious and requires knowing the derived prefix name.

**Suggested change:**

```python
def make_pytest_guard(
    *,
    repo_name: str,
    env_prefix: str | None = None,
    hints: list[str] | None = None,
    cooldown_s: float = 30.0,
    posture: str | None = None,          # ← NEW
) -> RuntimeGuard:
    derived_prefix = ...
    if posture and not os.environ.get(f"{derived_prefix}_POSTURE"):
        os.environ[f"{derived_prefix}_POSTURE"] = posture
    return RuntimeGuard(...)
```

This makes `make_pytest_guard(repo_name="MyRepo", posture="ci")` work
without the caller needing to know the derived prefix convention.

---

### 2. Add a `wsl_dev` posture preset

**Problem:** `ci` (1024 MB, 90% swap) is the most IDE-friendly existing preset,
but there is no named preset for developer WSL2 environments where a heavy IDE
(VS Code, JetBrains) permanently consumes 6–12 GB RAM with nothing to do about it.

**Suggested addition to `_PRESETS`:**

```python
# wsl_dev: tolerant of VS Code / JetBrains holding large resident RAM
# outside pytest's control.  Only raises pressure when swap is exhausted
# or the pytest process itself is starved to <256 MB.
"wsl_dev": (256, 97, 128, 99, 10),
```

The `self_inflicted_pct=10` means attribution will still correctly classify
pressure that *is* pytest's fault, just at a tighter self-responsibility band.

---

### 3. `preflight_check` `auto_intervene` pattern should be in make_conftest_content

The `make_conftest_content()` helper generates a minimal conftest stub.
The generated stub only calls `check_and_log()`, missing the more impactful
`oom_protect()` + `preflight_check(auto_intervene=True)` + background check
pattern.  Upgrading the template would improve every adopter's baseline
protection with no extra effort.

**Suggested generated pattern** (replace existing template):

```python
def pytest_sessionstart(session):
    _guard.oom_protect(score=-500)
    _guard.preflight_check(abort_on_critical=False, auto_intervene=True)
    _guard.start_background_check(interval_s=60)

def pytest_runtest_setup(item):
    _guard.check_and_log(stage=item.nodeid)

def pytest_sessionfinish(session, exitstatus):
    _guard.stop_background_check()
```

## Feedback from Airlines Manager Optimiser (2026-05-10)

> Added by cross-repo WSL crash resolution workflow.
> Source: /home/thomas_stening/Airlines Manager Optimiser

### Context

Airlines Manager Optimiser runs Selenium/Chrome automation against a live
browser game.  Under WSL2 swap pressure (SwapUsed 90–99%), the Chrome
renderer OOM-kills itself, producing a silent `WebDriverException` with an
empty message and raw `#0 0x... <unknown>` stack frames — very hard to
diagnose.

The existing `_check_memory_pressure()` helper used `psutil`, but
runtime-guard was already available in the project and reads `/proc`
directly without any extra dependency.

### Local implementation

`_check_memory_pressure()` in
`airlines_manager_optimizer/automation/live_route_demand.py` was rewritten
to use `RuntimeGuard.subprocess_safe()`:

```python
guard = RuntimeGuard(env_prefix="AIRLINES_MANAGER_OPTIMISER_GUARD", ...)
safe, reason = guard.subprocess_safe("Chrome", min_mb=500, stage="pre-chrome-audit")
return (not safe), reason
```

When memory is critical the function raises `RuntimeError` which is caught
by the existing `try/except` in `verify_current_schedule`, turning a silent
Chrome crash into a readable warning:
`⚠️ Live route-demand audit failed — Chrome launch skipped — system under memory pressure`

### Validation evidence

```
4 passed in 0.35s   (tests/test_live_route_demand.py)
178 passed in 1.78s (runtime-guard test suite)
```

Module-level convenience function confirmed working:
```
subprocess_safe: safe=True  reason=''         # ← normal conditions
subprocess_safe: safe=False  reason='HeavyProc launch skipped — MemAvailable=4185 MB < 999999 MB threshold'
```

### Suggested improvement implemented: `subprocess_safe()`

**Problem:** No first-class API existed for "is it safe to launch a heavy
subprocess right now?".  Callers had to combine `check()` + manual RSS
comparison, or reach for `psutil`.  This pattern appears in any project
that drives browser automation, Java VMs, or multi-process data pipelines.

**Implemented in runtime-guard:**

1. `RuntimeGuard.subprocess_safe(label, *, min_mb, stage)` — method on the
   guard instance.  Checks `mem_available_mb >= min_mb` first (hard floor),
   then calls `check()` for the critical threshold.  Returns `(bool, str)`.

2. `subprocess_safe(label, *, min_mb, env_prefix)` — module-level convenience
   wrapper; no need to instantiate a guard.  Added to `__all__`.

**Usage from any repo:**

```python
from runtime_guard import subprocess_safe

safe, reason = subprocess_safe("Chrome", min_mb=500)
if not safe:
    raise RuntimeError(f"Skipping Chrome — {reason}")
```

### Additional suggestions (not yet implemented)

**4. `env_prefix` mismatch between `make_sitecustomize_content()` and `make_pytest_guard()`**

`make_sitecustomize_content()` hardcodes `env_prefix='RUNTIME_GUARD'`,
while `make_pytest_guard()` auto-derives `REPO_NAME_GUARD`.  In this repo:
- `sitecustomize.py` reads `RUNTIME_GUARD_MIN_MEM_AVAILABLE_MB`
- `conftest.py` guard reads `AIRLINES_MANAGER_OPTIMISER_GUARD_MIN_MEM_AVAILABLE_MB`

This means tuning one env var does not tune the other.  Either:
- `make_sitecustomize_content()` should accept `repo_name=` and derive the
  same prefix, or
- Both helpers should default to `RUNTIME_GUARD_` and accept explicit override.

**5. Chrome/Selenium process recognition in `wsl_system_report()`**

When top-RSS consumers include `chrome`, `chromium`, or `chromedriver`, the
report's "Next Steps" section could call them out explicitly:

```
⚠️  Browser processes detected in top consumers:
   chrome (3 × ~350 MB each = ~1 GB).  These are the likely swap cause.
   Quit all browser sessions before running Selenium tests.
```

This would have made the original WSL crash much faster to diagnose.

## Feedback from runtime-guard workspace (2026-05-11)

### Crash investigation summary

- VS Code/Copilot sessions restarted repeatedly without explicit stack traces in chat debug logs.
- WSL host was under sustained memory pressure during the crash window (`~17 GiB / 19 GiB` used, low free memory, high Node/Chrome RSS).
- Most top consumers were VS Code extension host Node processes, with additional browser and Python memory pressure.

### Local implementation attempt

- Added `scripts/wsl_preflight.py` as a lightweight gate before launching heavy subprocesses.
- The script uses `RuntimeGuard.subprocess_safe(...)` and exits non-zero when launch risk is high.
- Designed for CI checks and developer workflows in WSL where browser/Node processes can cause hidden pressure spikes.

### Suggested improvement for runtime-guard core

Add a first-class CLI command in `runtime-guard` itself:

```text
runtime-guard --preflight-subprocess --label Chrome --min-mb 500
```

Why:
- Removes need for repository-local wrapper scripts.
- Standardizes exit codes and JSON output for orchestration tooling.
- Makes WSL crash prevention discoverable next to existing `--check` and `--report` workflows.

## Feedback from runtime-guard workspace (2026-05-11, host+guest crash triage)

### Context

After a VS Code WSL crash, investigation was run across both guest and host:

- Guest showed severe memory contention (`MemAvailable` near 1.0-1.6 GiB, swap in 70-93% range)
- Memory PSI showed active stalls (`full avg10` and `some avg10` elevated)
- Top RSS consumers were mostly VS Code extension host and language-server Node processes
- Host memory remained healthier than guest, confirming the immediate crash pressure was guest-local contention

### Local implementation attempt

`scripts/wsl_preflight.py` was extended with host+guest crash diagnostics:

- New mode: `--diagnose-crash`
- New gate: `--fail-on-risk {none,high,critical}`
- New outputs:
    - Guest RAM/swap + PSI (`/proc/meminfo`, `/proc/pressure/memory`)
    - Host RAM/virtual memory via PowerShell CIM
    - Drift fields between guest and host memory views
    - Computed risk level (`low|moderate|high|critical`) and likely causes/prevention actions

### Validation evidence

- Unit tests for parser/risk logic:
    - `tests/test_wsl_preflight.py`: `3 passed`
- Live diagnostics output (same environment):
    - `risk_level=critical`, `risk_score=5`
    - Causes identified: high guest swap + elevated PSI stalls

### Improvement suggestion to feed back into runtime-guard

Add a first-class library function and CLI mode for crash triage so adopters do not need script-local logic:

```text
runtime-guard --diagnose-wsl-crash --json --fail-on-risk high
```

Suggested API shape:

```python
from runtime_guard import diagnose_wsl_crash

report = diagnose_wsl_crash()
assert report["risk_level"] in {"low", "moderate", "high", "critical"}
```

Benefits:
- Consistent risk model and thresholds across repos
- Shared, testable host+guest diagnostics in core package
- Better operator UX for post-crash RCA and prevention automation
